# Webラジオサービス「ニュース・コネクト・シスターズ！」仕様書

## 1. サービス概要

AIキャラクター（しっかり者のお姉さんと、すこしお馬鹿な妹）が、最新のニュースをリスナーと対話しながら解説する、参加型のWebラジオサービスです。

**開発方針**: いきなり全自動化せず、ローカルで動く最小構成から始め、段階的に自動化・機能を足していく。

---

## 2. サービス名

**ニュース・コネクト・シスターズ！**（Connect Sisters）

## 5. 技術スタック

コストを抑えつつ、**重い処理（音声合成）をクラウドCIから切り離す**構成を採用する。

| レイヤー | 採用技術 | 理由 |
|----------|----------|------|
| フロントエンド | 静的HTML + CSS + JS | 依存少・デプロイが単純 |
| ホスティング | GitHub Pages | 無料・GitHub 連携が容易・Phase 1 に最適 |
| 定期実行 | GitHub Actions | 無料枠で台本生成など軽い処理に限定 |
| ニュース取得 | Python 3 + `feedparser`（RSSのみ） | シンプル・APIキー不要・Search APIは不要 |
| 台本生成 | Google Gemini API（`gemini-2.0-flash` 等） | 無料枠が充実・掛け合い生成に向く |
| 音声合成 | VOICEVOX Engine（ローカル → 後に自前VPS） | キャラ声の再現性。Actions上での実行は避ける |
| データ管理 | リポジトリ内の JSON / Markdown | DB不要で Phase 0 から使える |
| コメント（Phase 4〜） | Google フォーム + スプレッドシート + Apps Script | 無料・非エンジニアでも運用しやすい |
| 音声ストレージ | Git（最新1本）→ Cloudflare R2（アーカイブ、Phase 5〜） | リポジトリ肥大化を防ぐ |
| 秘密情報 | GitHub Secrets / ローカル `.env`（gitignore） | APIキーの安全な管理 |

### アーキテクチャ概要（目標形）

```
[RSS] → [GitHub Actions: 台本生成] → [台本 artifact]
                                              ↓
                                    [VOICEVOX: 音声合成]
                                    （ローカル or VPS）
                                              ↓
                              [Git コミット: latest.mp3 + script.md]
                                              ↓
                                    [GitHub Pages: 公開]
```

## 6. フェーズ別リリース計画

各フェーズは **前フェーズが安定してから** 次へ進む。機能は「そのフェーズの Done 条件」を満たした時点で完了とする。

---

### Phase 0: ローカル PoC（最初の1本を作る）

**目的**: パイプライン全体を手元で1回流し、品質と所要時間を把握する。

| 項目 | 内容 |
|------|------|
| 自動化 | なし（すべてローカル実行） |
| ニュース | 固定RSS 2〜3ソースから手動実行スクリプトで取得 |
| 台本 | Gemini API で掛け合い台本を生成 |
| 音声 | ローカルの VOICEVOX Engine で合成 |
| 公開 | 静的HTMLに `latest.mp3` を置き、`git push` で GitHub Pages に自動デプロイ |
| コメント | なし |

**Done 条件**
- [ ] ローカルスクリプト1本で「RSS → 台本 → 音声」まで完走できる
- [ ] 生成物をサイトで再生できる
- [ ] 1エピソードの生成にかかる時間・コスト（API使用量）を記録した

---

### Phase 1: MVP 公開（最新1本の Web ラジオ）

**目的**: 最小限の公開サイトを立ち上げ、第三者が聴ける状態にする。

| 項目 | 内容 |
|------|------|
| 自動化 | 台本生成のみ GitHub Actions（`workflow_dispatch` = 手動トリガー） |
| ニュース | Actions 上で RSS 取得 → Gemini で台本生成 |
| 音声 | **ローカルで VOICEVOX 合成** → 手動コミット |
| 公開 | `public/latest.mp3` + `public/script.md` + `public/status.json` をコミット → GitHub Pages 自動デプロイ |
| サイト | プレイヤー・台本表示・「準備中」表示（`status.json` 参照） |
| コメント | なし |

**Done 条件**
- [ ] Actions のボタン1つで台本が生成される
- [ ] 手動で音声をコミットするとサイトに反映される
- [ ] 生成失敗時、前回の音声がそのまま表示される

---

### Phase 2: 定期配信（毎朝の半自動運用）

**目的**: 毎日決まった時間に台本が自動生成され、運用ルーティンが回る。

| 項目 | 内容 |
|------|------|
| 自動化 | GitHub Actions を cron で毎朝実行（例: 平日 6:00 JST） |
| ニュース | 同上 + 取得失敗時はキャッシュ or スキップして通知 |
| 音声 | 引き続きローカル合成 → コミット（朝のルーティンとして定着） |
| 公開 | Phase 1 と同様 |
| 通知 | Actions 失敗時にメール or Issue 自動作成 |

**Done 条件**
- [ ] 1週間、平日に台本が自動生成される
- [ ] 失敗時に気づける仕組みがある
- [ ] 入力トークン数が Gemini 無料枠内に収まっている

---

### Phase 3: 音声合成の自動化

**目的**: 音声生成の手作業をなくし、エンドツーエンドの自動配信を実現する。

| 項目 | 内容 |
|------|------|
| 自動化 | 台本生成 → 音声合成 → コミット → デプロイまで一連のフロー |
| 音声合成 | **Oracle Cloud Always Free 等の VPS** に VOICEVOX Engine を常駐させ、HTTP API で呼び出す（GitHub Actions 上では実行しない） |
| 代替案 | VPS が難しい場合は、Actions 完了後にローカル常駐スクリプトが artifact を取得して合成する半自動方式でも可 |
| 音声形式 | MP3（128kbps）に変換してコミットし、リポジトリ容量を抑える |

**Done 条件**
- [ ] 人の手を介さず、新しいエピソードがサイトに載る
- [ ] 1エピソード生成の全体所要時間が許容範囲内（目安: 30分以内）

---

### Phase 4: リスナー参加（コメント読み上げ）

**目的**: リスナーとの双方向性を加え、サービスの核となる参加型体験を実現する。

| 項目 | 内容 |
|------|------|
| コメント投稿 | Google フォーム（ニックネーム + コメント本文） |
| 保存 | Google スプレッドシート（`status`: `pending` / `used` / `rejected`） |
| 取得 | Actions 実行時に **1回のバッチ読み取り**（未処理を最大3件） |
| 台本 | ニュース本編の後に「リスナーコメントコーナー」を追加 |
| モデレーション | NGワードフィルタ + 手動で `rejected` に変更できる運用 |

**Done 条件**
- [ ] フォームから投稿されたコメントが次回エピソードで読み上げられる
- [ ] 不適切なコメントを配信前に除外できる

---

### Phase 5: 運用安定化・アーカイブ

**目的**: 長期運用に耐える品質・体験を整える。

| 項目 | 内容 |
|------|------|
| アーカイブ | 過去エピソードを Cloudflare R2（無料枠）に保存、サイトに一覧ページ |
| リポジトリ | `latest.*` のみ Git 管理、過去分は外部ストレージ |
| 品質 | オープニング/エンディングの音声キャッシュ、テンプレート台本のフォールバック |
| 法務 | 免責事項・プライバシーポリシー・出典表示のページ |
| 監視 | `status.json` に `ok` / `degraded` / `failed` を記録 |

**Done 条件**
- [ ] 過去10回分以上をサイトから聴き返せる
- [ ] 法務・免責の最低限のページがある

## 9. 運用フロー

### Phase 1〜2（半自動）

1. GitHub Actions が起動（手動 or cron）
2. RSS からニュースを取得（最大3件）
3. Gemini API で掛け合い台本を生成
4. 台本を artifact として保存 / コミット
5. **人がローカルで VOICEVOX 合成 → `latest.mp3` をコミット**
6. GitHub Pages が自動デプロイ
7. サイトに最新エピソードが反映

### Phase 3以降（全自動）

1. 上記 1〜4 に加え、VPS 上の VOICEVOX API で音声合成
2. MP3 + 台本 + `status.json` を自動コミット
3. Phase 4: スプレッドシートから未処理コメントを取得し、台本に組み込み
4. 使用済みコメントの `status` を `used` に更新

### 異常系

| 状況 | 対応 |
|------|------|
| RSS 取得失敗 | 前回キャッシュを使用、またはその日はスキップして `status: degraded` |
| Gemini API 失敗 | リトライ1回 → 失敗なら配信スキップ、前回音声を維持 |
| 音声合成失敗 | 台本のみ公開（テキストラジオモード）or 配信スキップ |
| コメント0件 | コメントコーナーを省略して通常配信 |

## 10. コスト管理

| フェーズ | 想定月額 |
|----------|----------|
| Phase 0〜2 | **0円**（Cursor API 無料枠 + GitHub Pages 無料枠） |
| Phase 3〜 | **0円**（Oracle Cloud Always Free VPS を利用する場合） |

### 無料枠を守るための制約

- ニュースは RSS のみ（Search API は使わない）
- 1エピソードあたりニュース3件・コメント3件まで
- 記事本文は渡さず「タイトル + 要約 + URL」のみ Gemini に入力
- 音声は MP3 圧縮、Git には `latest` のみ保持
- 配信は平日のみ（必要に応じて調整）

## 11. リスクと対策

| リスク | 対策 |
|--------|------|
| VOICEVOX が GitHub Actions で動かない | Phase 3 までローカル合成。以降は VPS に切り出す |
| Gemini 無料枠超過 | 入力圧縮・件数上限・平日のみ配信 |
| Git リポジトリ肥大化 | `latest` のみコミット。Phase 5 で R2 に移行 |
| スプレッドシート API 制限 | 1日1回のバッチ読み書きのみ |
| AI の事実誤認 | 要約＋出典リンク表示。断定口調を避けるプロンプト設計 |
| ニュース著作権 | 本文朗読はせず、要約と出典明示に留める |

---

## 12. 法務・コンテンツ方針

- 本サービスは AI 生成コンテンツであることをサイトに明記する
- ニュースは各メディアの RSS を利用し、出典 URL を台本・サイトに表示する
- コメントは投稿時に利用規約への同意を求める
- 個人情報（本名・連絡先等）の投稿をフォームで禁止する
